# IPMI Simulator
Простой IPMI симулятор на основе [ipmi_sim](https://github.com/cminyard/openipmi), собранный в контейнер на базе Alpine.

## Цифровой двойник
На данный момент реализован минимальный пример того, как симулятор может быть использован в рамках цифрового двойника. Предполагается такой сценарий:
1. Необходимо проверить навыки студентов по работе с протоколом `IPMI`;
2. Поднимается n-e количество контейнеров, по 1 на студента (на данный момент не реализовано, можно просто реализовать засчет `docker-compose`, `kuber`...);
3. В каждом из контейнеров выполняется случайный `.sh` скрипт сценария, который просто инжектит в симулятор некую проблему (Сейчас реализован только сценарий перегрева ЦПУ. Скрипт сценария в данном случае просто подменяет значение температуры на 95°C. Симулятор в свою очередь это подхватывает, генерирует события тревоги);
4. Студенту необходимо засчет команд `ipmitool` определить, есть ли в системе проблема и какая;
5. После определения проблемы студент предпринимает управляющие действия, если того требует ситуация (к примеру, в случае перегрева ЦПУ, ожидается что студент выполнит команду power reset или power off/on);
6. В конце студент запускает `.sh` скрипт `verdict.sh`, где в интерактивном режиме выбирает: какую проблему в системе он обнаружил.
7. `verdict.sh` вызывает `analyze.sh`, который определяет оценку студента по следующей формуле: `S = 0,4*V + 0,4*N + 0,2*E` (S - оценка, V - вердикт студента по проблеме (0/1), N - были ли произведены управляющие действия, если таковые требовались (0/1), E - доп. метрика, оценивает, произвел ли пользователь слишком много аналищирующих команд, слишком мало или в пределах разумного). В данный момент оценка выводится студенту сразу, можно изменить и собирать оценки со всех контейнеров для разных студентов допустим в `.json` файл
8. Упражнение завершено.

Еще раз, это **минимальный** пример, его можно доработать: добавить сценариев, добавить оркестрацию для множества контейнеров сразу, собирать данные по оценках от всех контейнеров в один файл, запускать в одном контейнере последовательно несколько упражнений и так далее.

## Проверка реализованного сейчас примера:
Необходимо получить образ контейнера. Для этого рекоммендуется спуллить образ из `Docker hub`:
```bash
docker pull nissanissa/ipmi-sim:latest
```
Также можно собрать образ из исходников в данном репозитории:
```bash
docker build -t ipmi-sim:latest .
```
Далее нужно запустить контейнер:
```bash
docker run -d --rm --name ipmi-sim -p 623:623/udp ipmi-sim:latest
```
Затем зайти в его оболочку:
```bash
docker exec -it ipmi-sim bash
```
Для начала выполнения упражнения запустите `inject_cpu_overheat.sh`:
```bash
./inject_cpu_overheat.sh
```
Теперь в рамках сценарий вы "должны установить проблему и устранить ее". Предполагается, что студент выполнит это примерно с помощью такой последовательности команд:
```bash
ipmitool -H 127.0.0.1 -C 3 -I lanplus -U ADMIN -P ADMIN sel list
ipmitool -I lanplus -H 172.17.0.2 -p 623 -U ADMIN -P ADMIN -C 3 sensor reading "CPU Temp"
ipmitool -I lanplus -H 172.17.0.2 -p 623 -U ADMIN -P ADMIN -C 3 power reset
ipmitool -I lanplus -H 172.17.0.2 -p 623 -U ADMIN -P ADMIN -C 3 sensor reading "CPU Temp"
```
После этого студент хочет завершить выполнение упражнения и вызывает `verdict.sh`:
```bash
./verdict.sh
```
В интерактивном режиме он вводит номер обнаруженной проблемы. После этого упражнение завершено, студент видит свою оценку.

С помощью записи в файл конкретных значений можно сымитировать динамическое изменение данных сенсора. В данном симуляторе это делается с помощью скрипта `dynamic.sh`. Если необходимо внести изменения в логику динамического изменения данных, нужно написать скрипт, который будет вносить изменения в файлы, указанные в `sim.emu` и собрать образ контейнера.
